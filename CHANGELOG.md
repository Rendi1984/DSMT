## 3.29.24
**A full live-console pass by the user turned up 10 issues in one sitting. Root-caused and fixed all of them except two (flagged below, no code bug found - need live re-test with better error surfacing).**
- **Settings -> General/Backup showed a stale/wrong LDAP server value, forever** (`index.html`, `DSMT_Api.ps1`): there was no `loadConfigLive()` - sign-in loaded health, DB info, and alerts, but never re-fetched `/api/config`, so General/Backup kept showing the client-side demo placeholder until someone manually used "Import config". Worse, `GET /api/config` and `POST /api/config` didn't even read/write the same store: GET returned only the SQL `dbo.Config` table, POST wrote only `config.json` - so even a correct reload would have shown incomplete data. Fixed both: `GET /api/config` now merges `config.json` (LDAP server, base DN, domains, AD Connect server) with the SQL-backed fields (access group, contractor OUs); `POST /api/config` now writes to both stores symmetrically; and the console calls a new `loadConfigLive()` right after sign-in (and again when opening the General tab) to keep the fields live.
- **Database tab "Backup now" button did nothing**: `apiFetch` (`index.html`) always sent `Content-Type: application/json`, even on bodyless POST requests - Pode's body parser choked on the empty body before the route handler ever ran, producing a generic failure with no useful message. Fixed: the header is now only added when a request actually has a body.
- **Access & Permissions tab UX** (`index.html`, per explicit user request): the group-mapping "add new" row's group picker is now a free-text LDAP group name input (role stays a dropdown next to it); the "Local accounts" card moved directly under "Group -> role mapping" (was previously separated by "Local default administrator"). No unified Save button added - each action already auto-saves individually by design.
- **Event Viewer returned no results, local or remote** (`Diagnostics.psm1`): `Get-RemoteEvents` always called `Get-WinEvent -ComputerName`, which forces the legacy RPC remoting path and requires the "Remote Event Log Management" firewall rule group even when querying the API's own host. Added `Test-IsLocalMachine` and skip `-ComputerName` entirely for a self-query.
- **Diagnostics -> Domain Controllers showed almost everything amber/unreachable** (`Diagnostics.psm1`): `Test-DcServices` set a sticky flag on the *first* service-probe failure for a host and then marked every remaining service on that host `unknown`, regardless of its real state. Each service is now probed independently, and the same local-machine RPC-avoidance fix from Event Viewer applies here too (`Get-Service -ComputerName` has the same self-query firewall requirement).
- **Alerts can now deep-link to the relevant Settings tab** (`index.html`, new feature): clicking a setup/certificate alert (e.g. "setup not finished", an expiring certificate) jumps straight to Access & Permissions or Certificate Authority settings instead of just showing the alert text.
- **The whole console is now responsive** (`index.html`, new feature): below ~860px width the sidebar collapses behind a hamburger button in the header; opening it overlays the sidebar with a dismissible backdrop, and it auto-closes after picking a nav item or resizing back to a wide window.
- **`*Live` list loaders no longer leave stale/undefined state on a failed fetch** (`index.html`): e.g. `loadUsersLive` now resets to an empty list and shows the real error message on failure, instead of silently leaving `undefined` behind (the likely source of the `[dc-runtime] {{ }} never resolved` console warnings).
- **Uninstall now removes everything by default** (`Uninstall-DSMT.ps1`, per explicit user request): the Windows service, install folder, IIS site + webroot, and firewall rules are all removed unless you pass the new `-LeaveIIS` switch (keeps the IIS site/webroot/firewall rules for a fast reinstall). Logs and the SQL database are still kept unless `-RemoveLogs`/`-RemoveDatabase` are passed, same as before.
- **DL Groups "Generate" and "Create user" - no code bug found.** Client/route/module wiring checked out correctly for both; these need a live re-test now that failures surface a real error message instead of failing silently, to reveal whatever the actual live-environment cause is (AD permissions, group/OU name mismatch, etc.).
- Hot-swap: `index.html` -> IIS webroot then Ctrl+F5; `DSMT_Api.ps1` -> `<InstallDir>\server\` then `Restart-Service DSMT-Api`; `Diagnostics.psm1` -> `<InstallDir>\server\modules\` then `Restart-Service DSMT-Api`; `Uninstall-DSMT.ps1` -> only relevant on your next uninstall run.

## 3.29.22
**Real bug (repeated `400 Bad Request` on Settings -> General -> Save changes)**: `Save-Config`'s helper in `DSMT_Api.ps1` defaulted its file path to the bare script-level `$cfgPath` variable, which - exactly like `$Config` in the 3.29.20 fix - is not visible inside a Pode route's own runspace. Every route calling `Save-Config` without passing a path (`/api/config`, `/api/ca/config`, `/api/db/config`, and the already-fixed `/api/setup/save`) was silently getting `$Path = $null`, so `Set-Content` threw and every Settings-tab save returned a generic `400`. Fixed: `Save-Config` now requires the caller to pass `-Path $using:cfgPath` explicitly (the default remains for top-level script callers), and all 4 call sites now do.
- **New feature**: added a first-time-setup indicator directly on the affected Settings tab, not just the alerts bell. The **Access & Permissions** tab label now shows a warning marker while setup tasks are pending (default admin password still in use, or no LDAP group yet mapped to System Administrator), and the tab's content opens with a red callout listing exactly what's left to do. Both clear automatically the moment each task is completed - same underlying signal the alerts bell already used, just also visible right where you need to act.

## 3.29.21
**Good news first: 3.29.20's fix worked** - `dsmt-request.log` confirms the setup wizard's `test-server`/`create-db` succeeded and, for the first time, `/api/health` returned a populated (non-SETUP-MODE) response, meaning `/api/setup/save` finally persisted `config.json` and the database was created. The remaining failure (`POST /api/auth/login` -> `500`) is a **real SQL permissions issue in the environment**, not this code - but it exposed a genuine robustness gap that's fixed here:
- **`Write-Audit` had no error handling at all** (`Db.psm1`) - every route calls it fire-and-forget (login, user actions, CA actions, ...) expecting it never to fail. When SQL briefly rejects a connection (as happened here), `Write-Audit` threw, crashing whatever unrelated route called it with a raw, unhelpful `500` that masked the real error. It's now wrapped in try/catch and logs failures to `dsmt-sql.log` instead of ever breaking the caller.
- **`/api/auth/login`'s session-creation step** (`DSMT_Api.ps1`) was outside the route's existing try/catch (which only covered the initial `Invoke-SignIn` call) - a SQL failure creating the session row now returns a readable `502` instead of an empty `500`.
- **The actual environment issue, for the record**: the error log showed `Login failed for user 'NT AUTHORITY\SYSTEM'` - when SQL Server and the API run on the *same* machine and the connection uses Windows Integrated auth, a same-machine ("loopback") SSPI connection is a well-known case where Windows presents the service's identity as `NT AUTHORITY\SYSTEM` to SQL Server, regardless of whether the service account is NetworkService, LocalSystem, or otherwise - not the computer account you'd normally expect for a *remote* connection. Fix: either grant that login rights on `DSMTOOL`, or (simpler when SQL and the API share a machine) switch `Database.Auth` to `SQL` and use a SQL Server login instead of Windows auth. See the updated Deployment Guide / README for the exact commands.

## 3.29.20
**This is very likely the actual root cause of the browser setup wizard never completing, across every attempt in this whole session.** `POST /api/setup/save` (`DSMT_Api.ps1`) read and wrote `$Config` as a bare variable - `$Config.Database.Engine = $d.Engine`, etc. Pode route scriptblocks run in their own runspace; every other route in this file captures the outer config via `$cfg = $using:Config` first (a pattern this project has hit repeatedly - see 3.29.13, 3.29.15, 3.29.18). This route was the one place that skipped it, so `$Config` was `$null` inside the route, and the property assignments either silently did nothing or threw before `Save-Config` was ever reached. The wizard's `test-server` and `create-db` steps could succeed all day, but `save` could never actually persist the SQL connection or seed the local administrator - explaining why "Install" never finished no matter how many times it was retried, on `localhost`, a hostname, or otherwise.
- Fixed: captures `$using:Config` into a local `$cfg`, operates on that, and passes it explicitly to `Test-SetupComplete -Cfg $cfg` and `Save-Config -Cfg $cfg` (both already support that parameter - other routes call them the same way).
- Hot-swap: `DSMT_Api.ps1` -> `<InstallDir>\server\DSMT_Api.ps1`, then `Restart-Service DSMT-Api`. No reinstall needed.
- Also added a dedicated **Remote access** walkthrough to `README.md` (new section 6b) and `Deployment_Guide.html` (new "Accessing by server name, or from another machine" section, linked in the TOC before the reverse-proxy section): both the console and API already listen on all interfaces, so reaching DSMT by hostname or from a different machine only needs the right URL, firewall scope, and DNS - the most common mistake is using `localhost` from a remote browser, which resolves to the client, not the server. Docs-only addition, no further code change.

## 3.29.18
**One real server bug (sign-in HTTP 500) + two console 404 noise fixes, all found from the user's `dsmt-error.log` and browser console:**
- **Sign-in returned HTTP 500** (`Db.psm1`): `Invoke-Sql` bound *every* SQL parameter as `NVarChar(max)`. Plain column INSERT/WHERE usage survives via implicit conversion, but the session INSERT passes `@h` to `DATEADD(hour, @h, ...)` - a typed T-SQL function argument - and SQL Server rejects it outright: `Argument data type nvarchar(max) is invalid for argument 2 of dateadd`. So login authenticated successfully and then blew up creating the session row. Fixed with type-aware binding (bool -> Bit, int/long -> BigInt, datetime -> DateTime2, everything else stays NVarChar).
- **404 on `.image-slots.state.json` on every console load** (`index.html`): the logo image-slot helper (a design-time tool) always fetched its sidecar state file, which never exists on a deployed server. It now skips the fetch entirely when the design-tool host bridge is absent - slots just render as empty placeholders, no network request.
- **404 on `/favicon.ico`** (`index.html`): the page had no icon, so the browser's automatic favicon request 404'd on every load. Added a small inline SVG favicon (data URI, no external file).
- Hot-swap: `Db.psm1` -> `<InstallDir>\server\modules\` then `Restart-Service DSMT-Api`; `index.html` -> IIS webroot then Ctrl+F5.

## 3.29.17
**Root cause of "Install button does nothing" in the browser setup wizard, found from server request logs**: the re-entrancy guard in `setupPrimary` checked `this.state.installing`, but that flag is only set via an asynchronous `setState` inside `runSetupLive` - if the button fired more than once before that state update actually committed (rapid double-click, or any duplicate event), every invocation read the still-`false` value and proceeded, launching multiple **overlapping** `runSetupLive()` calls in parallel. Each overlapping call reset `installLog` and raced the others, so `/api/setup/test-server` and `/api/setup/create-db` fired dozens of times back-to-back (visible in `dsmt-request.log` as endless repeating pairs) while `/api/setup/save` was never cleanly reached - explaining both the "nothing happens" symptom and the runaway request log.
- Fixed with a synchronous, non-state instance flag (`this._setupBusy`) set the instant the button is pressed - independent of any state-commit timing - and cleared on every exit path (success, each failure branch, cancel, and the Demo-mode completion timeout).
- No server-side change needed - this was purely a client-side re-entrancy bug in `index.html`.

## 3.29.14
**Two more real bugs found from lab testing after 3.29.13, both blocking install/first-use:**
- **Duplicate CORS header on 401 responses**: the CORS middleware in `DSMT_Api.ps1` stamps `Access-Control-Allow-Origin: *` on every response using `Add-PodeHeader` (append), and `Write-401` stamped the same three headers again on top of it - producing `Access-Control-Allow-Origin: *, *`, which browsers treat as invalid and block outright ("Access-Control-Allow-Origin header contains multiple values"). Hit whenever an unauthenticated request reaches an authenticated route (e.g. testing `/api/db/test` before signing in). Fixed by switching all CORS header stamping from `Add-PodeHeader` to `Set-PodeHeader` (overwrite instead of append).
- **`Install-DSMT.ps1` IIS deploy step could throw and leave the site stopped**: `Start-Website` (WebAdministration/COM) can throw `The object identifier does not represent a valid object` (HRESULT 0x800710D8) immediately after `Remove-WebBinding`/`New-WebBinding` in the same session - a known stale-handle quirk in the IIS PowerShell provider, especially likely after re-running the installer against an existing site. Fixed by falling back to `appcmd.exe start site` when `Start-Website` throws.
- Reminder: the console must be opened via the IIS URL (`http://<server>:8080` or the reverse-proxy URL) - opening `index.html` directly as a `file://` path always fails CORS (`origin: 'null'`) regardless of server-side fixes, since the browser refuses to treat a local file as a valid CORS origin.

## 3.29.13
**Fixes the API failing to start on every fresh install** (console shows "Failed to fetch" / `ERR_CONNECTION_REFUSED` on every Live-mode call, even though `Get-Service DSMT-Api` reports "Running").
- **Root cause**: the `POST /api/users` (create user) route used `$using:Config.Directory.BaseDN` - a member-access chain directly on a `$using:` scoped variable. Pode's startup scope scanner (`Find-PodeScopedVariableUsingVariableValue`) cannot resolve this pattern on Windows PowerShell 5.1 and throws `Exception calling "ContainsKey"... Key cannot be null` while parsing **all** routes at server startup - before any request is even served. This crashes the whole Pode process immediately on every start.
- Because the API runs as a Windows service via a native wrapper that respawns the crashing child process every 3 seconds, `Get-Service` keeps reporting "Running" indefinitely while the actual HTTP listener never comes up - so every browser call fails with a generic network error, with no indication in the UI of what's wrong.
- Fixed by capturing `$using:Config` to a local variable first (`$cfg = $using:Config`) and then accessing `.Directory.BaseDN` on the local copy, matching the pattern already used everywhere else in the file.
- Also hardens both installers (`Install-DSMT.ps1`, `Install.ps1`) to automatically reserve the `http.sys` URL ACL (`netsh http add urlacl`) for the service's logon account when registering `DSMT-Api` - a separate, pre-existing gap that would otherwise cause the same "service shows Running but nothing listens" symptom for any account other than a fully-elevated one.
- **If you're on an already-broken install**: pull this fix, replace `DSMT_Api.ps1`, then `Restart-Service DSMT-Api` (no reinstall needed). Check `C:\Program Files\DSMT\logs\dsmt-service.log` to confirm the `Find-PodeScopedVariableUsingVariableValue` error is gone and the log shows the "listening on http://..." line.

This is now the only verification method proven to catch this bug class - the previous headless-DOM-text-grep check does not, since raw unexecuted script text appears in a DOM dump regardless of success or failure.

## 3.29.8
**Second critical fix, same incident as 3.29.7** - the console got past the JSON-unpack error but then failed with `Invalid regular expression: missing /` during class evaluation.
- **Root cause**: the CSV-export helper's regex `/[",\n]/` (used by `exportDl`/`exportPwExp` to decide whether a field needs quoting) had a **real newline character** baked into the regex literal itself, instead of the literal two-character escape sequence `\n`. This came from the original Python patch script that introduced the function, which used a single backslash where a doubled one was needed - so the character that landed in the file was an actual line break, not the text `\n`. A raw line break inside an unterminated JS regex literal is a syntax error, which crashed the whole class body evaluation (so nothing rendered at all, not just the CSV feature).
- **Fix**: corrected the encoding so the regex literal decodes to the literal two characters `\n`, matching its original intent (also match on embedded newlines when deciding to quote a CSV field).
- **Verification tightened further**: confirmed by decoding the affected string and checking it byte-for-byte (not just that `json.loads` succeeds), and by the same real render check as 3.29.7 (loading placeholder gone, no error-toast text, real sign-in screen present).
## 3.29.7
**Critical fix: the console failed to load entirely** ("Error unpacking: Bad control character in string literal in JSON at position ..."), breaking every install since this repo's edits began storing the app template as a JSON string.
- **Root cause**: `index.html`'s entire page template is stored as a JSON string value inside a `<script type="__bundler/manifest">`-style block, requiring every newline and double-quote within it to be escaped (`\n`, `\"`). Edits across recent releases inserted 378 real newline characters and 20 real double-quote characters directly (valid as plain HTML/JS, but each one breaks the enclosing JSON string) - the newlines came from routine multi-statement code insertions, the quotes from the CSV-export code (`exportDl`, `exportPwExp`) using un-escaped `"` characters inside single-quoted JS strings. The browser's `JSON.parse` of the template failed immediately, so the app never got past its own loading placeholder - it would have failed the same way on every release since the affected edits landed.
- **Fix**: re-escaped all 378 raw newlines and 20 raw quotes within the template's JSON string. Verified by parsing the extracted JSON string directly (not just checking brace/paren balance) and by confirming the loading-placeholder element is actually replaced and no "Error unpacking" text appears after the page runs - the previous verification method (grepping rendered DOM text) was unreliable here because the raw, not-yet-executed script content still appears in a DOM dump regardless of whether unpacking succeeded, so it never caught this.
- **Process fix**: `index.html` changes going forward are verified by decoding the template's JSON string with a real JSON parser, not just checking that expected text substrings appear somewhere in a headless DOM dump.

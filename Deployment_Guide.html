Directory Services Management Tool

Deployment & reference guide

Everything needed to stand up DSMT on a domain-joined server and understand what every screen actually does once it's live.

What DSMT is A browser console (index.html) that talks to an on-prem PowerShell REST API. The API is the only thing that touches Active Directory, SQL Server, your Certificate Authority and AD Connect — the console itself has no direct access to any of them.

LayerWhat it isTalks to
Consoleindex.html — a single offline HTML file, served by IISThe API only, over HTTPS + Bearer token
APIDSMT_Api.ps1 — PowerShell REST API (Pode), runs as the DSMT-Api Windows serviceActive Directory (LDAP/RSAT), SQL Server, AD Certificate Services (certutil), AD Connect server (PSRemoting), SMTP
DatabaseSQL Server, database DSMTOOLSessions, local accounts, role mappings, audit log, secrets, job history, config

The database does not hold your directory data — users, groups, certificates are always read live from AD / the CA. SQL is the console's own memory: who's logged in, who's allowed to do what, and what happened.

Before you start

Prerequisites

App server

  • Windows Server 2019/2022, domain-joined
  • Windows PowerShell 5.1 (built in)
  • Administrator / elevated session for install

SQL Server

  • Any edition — SQL Express is enough for this workload
  • Windows or SQL-login auth, reachable from the app server

Console client

  • Any modern browser (Chrome, Edge, Firefox, Safari)
  • No install, no extensions — index.html is fully self-contained

Optional integrations

  • Azure AD Connect server — only if using the Sync page
  • AD Certificate Services — only if using the CA page
  • SMTP relay — only if sending real test messages from Diagnostics

See Ports & firewall and Permissions below for exactly what each optional piece needs — skip any row for a feature you won't use.

Network

Ports & firewall

Two ports are yours to open for the app itself; everything else is what the app server needs to reach outward to do its job. Ports shown are defaults — all are configurable in config.json or the installer parameters.

* Default dynamic RPC range is 49152–65535 on modern Windows. On the target machine, enable the built-in firewall rule groups “Remote Service Management” (Diagnostics) and “Remote Event Log Management” (Event Viewer) instead of opening the range by hand — they cover the endpoint mapper and the negotiated port together.

http.sys needs an explicit reservation for non-admin service accounts Binding Api.Port over plain HTTP uses http.sys under the hood — a service account that isn't fully elevated (NetworkService, a domain service account, a gMSA) needs a URL ACL reservation to bind it (netsh http add urlacl url=http://+:8780/ user="<account>"). Both installers do this automatically since 3.29.14 when registering the service — if you registered the service another way, or moved it to a different account afterward by hand, run that command yourself or the API will silently fail to bind while the Windows service still shows "Running".

Minimal footprint Only enable what you'll actually use. A deployment that never touches Sync, CA, Diagnostics or Event Viewer needs nothing beyond SQL + LDAP/Kerberos/DNS to the DC.

Identity & access

Permissions

Two different identities are involved and are easy to conflate: the account you run the installer as (needs elevated, one-time rights) and the account the DSMT-Api service runs as afterward (needs narrower, day-to-day rights). Least privilege applies to the second one.

Runtime (the DSMT-Api service account — gMSA, domain service account, or NetworkService)

Security log is stricter Reading the Security event log in Event Viewer typically needs rights beyond Event Log Readers (e.g. membership that grants SeSecurityPrivilege) — expect Access Denied there until that's granted explicitly.

IIS application pool identity

Serves a single static file — no AD, SQL, or special rights needed beyond read access to the console's folder.

Install · Path A

Quick script recommended

One script does everything end to end: prerequisites, config.json, database + schema, local break-glass admin, the API Windows service, and the IIS-hosted console.

# from an elevated PowerShell, in the server\ folder
cd C:\DSMT\server
.\Install-DSMT.ps1

Anything you don't pass on the command line is asked for interactively. Only four things are truly required — everything else has a sensible default:

RequiredExample
-SqlServerSQL01 or SQL01\SQLEXPRESS
-LdapServerDC01.lab.local
-BaseDNDC=lab,DC=local
-LocalAdminPasswordprompted securely if omitted

Unattended example:

.\Install-DSMT.ps1 -SqlServer SQL01 -LdapServer DC01.lab.local `
    -BaseDN "DC=lab,DC=local" -Domains "lab.local" -ServiceAccount "LAB\svc_dsmt$"

Double-clicking Install.cmd at the project root does the same thing with a UAC prompt and a pause at the end — the easiest option if you're not comfortable with a terminal.

Zero-prompt installs Copy install-answers_sample.json to install-answers.json next to Install.cmd, fill in your values, and run — any value you provide is passed through and the installer stops asking for it. Leave a value blank to be prompted for just that one.

Useful switches: -SkipPrereqs, -SkipService, -SkipFrontend, -SkipDeploy, -NoOpen. All steps are idempotent — re-running is safe.

Fully offline / air-gapped app server Pass -Offline to skip every network call outright (no PSGallery, no Windows Update fallback) and fail immediately with the exact fix needed instead of hanging on a timeout. One-time prep on any internet-connected PC: Save-Module -Name Pode -Path .\vendor, then copy the resulting vendor\Pode\ folder into server\vendor\Pode\. If the target's Windows Feature source was stripped from the image, also pass -WindowsFeatureSource "D:\sources\sxs" (a mounted Windows Server ISO/ESD, or an extracted install.wim). The console itself has zero external references — nothing to prepare there.

Install · Path B

Browser-based setup wizard

Bootstrap the service without answering any SQL/directory/admin questions on the command line, then finish everything from the console itself.

  1. Bootstrap the service

    .\Install-DSMT.ps1 -SetupViaBrowser

    Installs prerequisites, deploys the files, registers the DSMT-Api service (which starts in SETUP MODE — no database yet), deploys the IIS console, and writes metadata to HKLM:\SOFTWARE\DSMT.

  2. Open the console and switch to Live

    On the sign-in screen, use the Demo / Live toggle. Live shows an API URL field — point it at http://<app-server>:8780.

  3. Run the setup wizard

    Click “Run the setup wizard” below the sign-in form. It asks the same questions the script would — SQL server, database name, domain controller, Base DN, and an administrator account (defaults to admin / admin, editable).

  4. The wizard does the real work

    Test server → create database + schema → save configuration and seed the break-glass administrator, driven by the same /api/setup/* routes the script uses.

  5. Sign in and finish the setup tasks

    Sign in with the administrator you just created. The alerts bell will show what's left — see First sign-in below.

Both installation paths are fully supported and produce an identical result. Pick whichever fits how the server was handed to you — script for scripted/unattended rollouts, wizard when someone without PowerShell access needs to finish the configuration.

Install · Path C

Manual, stage by stage

Useful when you want to run one stage at a time, or re-run just one step. All commands run from server\ in an elevated PowerShell.

  1. Prerequisites

    .\Install.ps1 -Prereqs

    Installs RSAT-AD-PowerShell and the Pode web framework (online from PSGallery, or offline from a bundled vendor\Pode folder).

  2. Configure

    copy config.sample.json config.json
notepad config.json

    Set Database, Directory (LdapServer, BaseDN), Sync.ADConnectServer, and CertificateAuthority.ConfigString (CA-HOST\CA-COMMON-NAME). For production, set Api.Protocol = https and supply a certificate.

  3. Create the database

    .\Install.ps1 -InitDb

    Creates the DSMTOOL database, all tables, and seeds default config + role mappings.

  4. Create the local break-glass admin

    .\Install.ps1 -SeedLocalAdmin -LocalAdminUser administrator

    Prompts for a password, stored as a PBKDF2-salted hash in SQL — never in plain text. This account signs in independently of AD, for first run or when LDAP is unreachable.

  5. Register the API service

    .\Install.ps1 -RegisterService -ServiceAccount 'LAB\svc_dsmt$'
# or, to run as NetworkService:
.\Install.ps1 -RegisterService

    Compiles and registers a native Windows service host (no NSSM dependency). Smoke test: Invoke-RestMethod http://localhost:8780/api/health.

  6. Point the console at the API

    Open index.html, sign in, go to Settings → Connection, switch Mode to Live, set the API base URL, Test connection, then sign out and back in.

Right after install

First sign-in & setup tasks

Sign in with the local administrator you created (default admin / admin from the browser wizard). The alerts bell (top right) tracks what's left to configure — it won't clear itself, so treat it as your post-install checklist:

action required

Default administrator password in use

Shown whenever the local admin is still on admin/admin. Change it in Access Control → Local accounts.

action required

Connect an LDAP admin group

Shown until a security group is mapped to the System Administrator role in Access Control → Role mappings — this is what lets your own domain admins sign in with their own accounts instead of the break-glass local one.

Both alerts clear automatically once resolved — no need to dismiss them by hand.

Demo vs. Live mode

The console always ships with Demo as a fully-working mode — realistic sample data, works completely offline, nothing is ever sent anywhere. It's meant to stay available for pre-deployment demos, training, or showing the product without touching a real environment.

Live mode routes every action through the real API against your real AD / SQL / CA. Switch between them any time from the Demo / Live toggle on the sign-in screen (API URL shown when Live is selected) — the choice and API URL persist across browser refreshes. A DEMO/LIVE badge in the top bar always shows which one is active.

Every screen, explained

Feature reference

Overview

read-only

Landing dashboard: system health tiles (SQL, LDAP, AD Connect sync), recent audit activity, and quick links. No configuration — it reflects whatever the other pages report.

GET /api/health

Azure Cloud Sync

requires setup

Triggers and monitors Azure AD Connect delta sync cycles on a remote AD Connect server, over PSRemoting.

To configure

Set Sync.ADConnectServer (Settings → General or config.json). The API's run-as identity needs PSRemoting rights and local admin on that box.

POST /api/syncGET /api/sync/status

DL Groups

Looks up a distribution/security group by name and lists its members (name, email, title, type, enabled state). Export downloads the result as a real .csv.

GET /api/dl/:group

User Management

Search AD users, create/reset/enable/disable/lock accounts, bulk actions on a selection, and a guided offboarding flow (disable → reset password → remove from all groups → hide from address book → move to a Disabled OU → audit entry).

To configure

The run-as identity needs delegated reset-password / enable-disable / create-user rights on the target OU(s). Offboarding's target OU comes from ContractorExpectedSupportOU in Settings.

GET /api/usersPOST /api/usersPOST /api/users/:sam/{reset,enable,lock}POST /api/users/bulkPOST /api/users/offboard

Password Expiry Report

Lists enabled users (excluding PasswordNeverExpires) whose password expires within a configurable window (30 days by default) — username, display name, OU, days left, expiry date. Computed per-user via msDS-UserPasswordExpiryTimeComputed, so fine-grained password policies (PSOs) are respected. Export downloads a .csv.

GET /api/passwords/expiring?days=30

Groups

Browse a security/distribution group's membership, add a member by SAM account name, or remove one.

GET /api/dl/:groupPOST /api/groups/:name/membersDELETE /api/groups/:name/members/:sam

Scheduled Jobs

requires setup

Enable/disable and manually run named jobs (e.g. “AD Connect Delta Sync”, “Disabled-User Cleanup”).

To configure

Each row corresponds to a Windows Scheduled Task with the same name on the app server — create those tasks first, then manage them here.

POST /api/jobs/:name/togglePOST /api/jobs/:name/run

Contractor Info

Looks up a contractor account and verifies it sits in the expected OU (admin vs. support), reading extensionAttribute10–15 for related server/system references.

To configure

ContractorExpectedAdminOU / ContractorExpectedSupportOU in Settings → General.

GET /api/contractor/:user

Audit Log

read-only

Every administrative action taken through the console — actor, action, target, result, source IP, timestamp — filterable by kind (user, password, access, cert, secret, db…). This is what makes Live mode safe to hand to an operations team.

GET /api/audit?kind=

Certificate Authority

Issued and pending certificate requests, approve/deny a request, revoke a certificate, publish the CRL, and back up the CA database. All actions run certutil -config "<host>\<CA name>" against the configured CA.

To configure

Settings → Certificate Authority: CA host (FQDN) and CA common name. The run-as identity needs Issue & Manage Certificates on that CA.

GET /api/ca/{certs,pending}POST /api/ca/{publish-crl,revoke,approve,deny,backup}

Event Viewer

new in 3.29

Reads a remote Windows server's event log over RPC — the same channel the graphical Event Viewer uses — so issues can be triaged without an RDP session. Filter by server, log (System/Application/Security/Setup), severity, time window, and free text or event ID.

To configure

The target server needs the “Remote Event Log Management” firewall rules enabled, and the API service account needs Event Log Readers (or admin) rights on it. The Security log typically needs elevated rights beyond that.

GET /api/events?server=&log=&hours=&levels=&q=

Diagnostics

read-only

Domain-controller health (discovers DCs three ways, then checks NTDS/DNS/Netlogon/KDC/W32Time/DFSR/ADWS), Exchange component health (any number of hosts), and a test-message tool that can either simulate or genuinely send through your SMTP server.

To configure

Targets need the “Remote Service Management” firewall rules enabled. A blocked ping does not, by itself, mean a host is reported unreachable — the service query is attempted regardless.

GET /api/diag/dcsPOST /api/diag/exchangePOST /api/diag/message

Access Control

Two independent checks happen at sign-in, and this page configures both — read it as two questions, not one:

  • Can this user sign in at all? Controlled by the Require security-group membership toggle + the one group picked in Access security group. When off, any valid domain credential can sign in.
  • What role do they get once they're in? Controlled by Group → role mapping — a table of LDAP groups, each mapped to a console role. A user matching no mapped group signs in as Read-only, even if they passed the check above.

Below that: the built-in local administrator's live status (enable/disable — reads from and drives the same account as the row in Local accounts, they can't drift apart), the local break-glass accounts list, and the MFA / SSO toggles.

To configure

Add at least one role mapping to System Administrator — this is the “Connect an LDAP admin group” setup task.

MFA is console-side only Enabling it prompts for a 6-digit code, but the API does not verify it server-side yet — don't rely on it as your only second factor in production.

GET/POST/DELETE /api/access/mappingsGET/POST /api/access/localPOST /api/access/local/:id/togglePOST /api/access/require-group

Data Source

The console's own SQL connection: host, port, database, auth mode, encryption. Two distinct test buttons: “Test server & list databases” checks the values currently in the form; “Test saved connection” checks what's actually saved in config.json — they are not redundant, they test different things.

POST /api/db/{test,config}GET /api/db/infoPOST /api/db/{migrate,backup}

Settings

General (LDAP server, Base DN, contractor OUs, sign-in domains), Connection (Demo/Live, API URL), Database, Certificate Authority, Access, and a config export/import (real JSON download / file-picker restore).

POST /api/config

Uninstall

.\Uninstall-DSMT.ps1 -Yes

Stops and removes the DSMT-Api service, deletes the install folder, and removes the HKLM:\SOFTWARE\DSMT registry key. Logs are kept by default (<InstallDir>\logs) so a failed install can still be diagnosed afterward — pass -RemoveLogs to delete those too. The IIS site, firewall rules and SQL database are all left in place unless explicitly requested:

SwitchEffect
-RemoveIisSiteAlso removes the IIS site and its webroot (C:\inetpub\dsmt by default)
-RemoveFirewallAlso removes the DSMT API/Console firewall rules
-RemoveDatabasePrompts for the SQL server/database, then permanently drops it — requires retyping the database name to confirm, even with -Yes
-RemoveLogsDeletes <InstallDir>\logs too (kept by default)
-YesSkips the confirmation prompt (unattended)

Full cleanup before a fresh reinstall, keeping the database and logs:

.\Uninstall-DSMT.ps1 -RemoveIisSite -RemoveFirewall -Yes
